# 0005 Knowledge Base Ingestion And Retrieval

Issue: M05 needs a knowledge-base source and retrieval path for leasing policies, FAQs, lease terms, application process details, and richer property descriptions. M04 added model-safe database read tools for structured property and unit facts, but the assignment also requires answers grounded in a separate knowledge base. This issue is being addressed now because M05 is the next incomplete milestone, and M07 grounded answer orchestration depends on a stable retrieval contract for non-database evidence.

Decision: Add a small repository-local knowledge base using committed Markdown source documents plus a deterministic lexical retriever. M05 will define a lightweight ingestion step that reads Markdown documents from a dedicated `data/kb/` source directory, splits them into bounded sections, assigns stable source IDs, and exposes retrieval through the existing `KnowledgeRetriever` protocol. Retrieval responses will return ranked snippets with source IDs, document titles, section headings, scores, and metadata suitable for later grounded answer composition. M05 will not add embeddings, vector databases, model calls, agent orchestration, natural-language answer generation, voice integration, or prospect writes.

Status: Accepted

Group: knowledge-retrieval

Assumptions: The brief says knowledge-base content is included, but the repository currently does not contain separate raw KB files, so M05 may create synthetic MVP content aligned with the seeded Lakeview Flats and Cedar Park Townhomes data plus a general leasing FAQ. One or two properties and a compact FAQ are enough for the assignment MVP. A deterministic lexical retriever is sufficient for the initial dataset and is easier to test than provider-backed semantic retrieval. Later milestones may replace or augment the retriever with embeddings without changing the orchestration-facing protocol.

Constraints: M05 must keep the knowledge base separate from the property database. It must not duplicate database-owned volatile facts as authoritative KB answers when the database already owns them, especially rent, availability, unit status, and exact unit attributes. It must return source-attributed snippets rather than final assistant prose. It must support explicit no-result behavior for unknown questions. It must run without credentials, network calls, or external services. It must not add write behavior, prospect capture, property-resolution state, full agent routing, model prompts, voice code, or a vector database service.

Positions: Position 1: Use committed Markdown KB source files and a deterministic lexical retriever. This keeps M05 small, inspectable, testable, and credential-free while still satisfying the need for source-grounded KB retrieval.

Position 2: Use JSON or YAML FAQ records only. This is easy to parse and test, but it is less natural for richer property descriptions and lease-policy content, and it can make future documentation updates more awkward.

Position 3: Add embeddings and a local vector index now. This may improve semantic recall later, but it adds model or embedding-provider decisions, index lifecycle complexity, and nondeterminism before the agent has an orchestration layer.

Position 4: Store KB content in SQLite alongside property data. This would simplify local persistence, but it weakens the assignment's separation between database facts and the knowledge base and increases schema scope for a content problem.

Position 5: Defer KB implementation until M07 and let the orchestrator call a fake retriever. This avoids immediate implementation work, but it postpones a required source of grounding and makes M07 too broad.

Argument: Position 1 is recommended because the project needs a real KB retrieval surface before answer orchestration, but it does not yet need semantic infrastructure. Markdown source files are easy to review in a pull request, can hold both property descriptions and general policy content, and work well as the durable source of truth for a small assignment dataset. A deterministic lexical retriever can be implemented with standard library tokenization, title/heading boosts, bounded snippets, and conservative scoring.

The main trade-off is recall: simple lexical retrieval will miss some paraphrases that embeddings could catch. That limitation is acceptable for M05 because the current dataset is intentionally small and the later orchestrator can still handle unknowns gracefully. The retriever should prefer transparent behavior over pretending to understand every query. If later evaluation scenarios show poor recall, an embeddings-backed retriever can supersede this ADR or be introduced behind the same protocol.

M05 should also avoid creating conflicts between the database and KB. The database remains authoritative for unit-level facts such as rent, availability, parking, view, bedrooms, bathrooms, square footage, pet policy fields attached to units, and status. KB documents may describe policies, application steps, deposits, lease terms, neighborhood notes, and broad property descriptions, but answer orchestration must later define precedence when DB and KB evidence overlap. For this milestone, retrieval should surface snippets and metadata only.

Implications: M05 implementation should add KB source documents, an ingestion/retrieval module, typed retrieval result DTOs if the existing protocol is not sufficient, and focused tests. Tests should cover FAQ retrieval, property-specific policy or description retrieval, source IDs and section headings, ranking behavior, snippet bounds, empty or unknown queries, and deterministic behavior without credentials. The architecture documentation should describe Markdown KB source, ingestion, retrieval, and source attribution. Requirements traceability should record that M05 adds concrete evidence for FR-03, FR-05, FR-10, NFR-03, and NFR-06, while not yet satisfying full answer orchestration or voice behavior.

Related decisions: ADR 0001 established Python/FastAPI, `uv`, `src/` layout, `pytest`, `ruff`, and `mypy`. ADR 0002 established provider and repository protocols, including the `KnowledgeRetriever` interface and deterministic fakes. ADR 0003 established SQLite persistence and seeded database facts. ADR 0004 established structured database read tools and evidence records. Later ADRs for property resolution, grounded answer orchestration, safe prospect capture, and evaluation will depend on or refine the retrieval evidence shape introduced here.

Related requirements: FR-03 requires answers from the knowledge base for policies, application process, lease terms, FAQs, and richer descriptions. FR-05 requires graceful handling of unknown questions. FR-10 requires sensible agent tools for database reads, knowledge-base retrieval, and database writes. NFR-03 requires clean, maintainable structure. NFR-06 requires automated verification for core behavior. EVAL-02 evaluates knowledge grounding, and EVAL-03 evaluates agent tool design and safety.

Related principles: Keep KB content reviewable and source-attributed. Return evidence snippets, not final answers. Prefer deterministic retrieval until orchestration and evaluation prove a need for embeddings. Keep database-owned facts authoritative in the database. Do not add external services before the milestone requires them. Preserve a narrow retriever interface so later retrieval implementations can be swapped behind the same contract.
