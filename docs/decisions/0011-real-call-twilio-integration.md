# 0011 Real Call Twilio Integration

Issue: M11 needs to expose the completed M10 voice pipeline through a real voice transport. M10 can already accept one audio turn, transcribe it, call the shared conversation session, compose a grounded model-backed spoken response, and synthesize assistant audio, but there is no inbound phone-call path, websocket transport, Twilio call control, public callback setup, or manual real-call verification path. The brief identifies a real inbound call as the ideal evidence and allows a browser voice loop only if telephony setup is blocked. The user has clarified that M11 should use an actual call and Twilio, so the browser alternative is no longer the primary implementation path.

Decision: Implement M11 as a Twilio-first real inbound call integration over the existing M10 voice pipeline. Add a small FastAPI telephony surface that answers Twilio inbound voice webhooks with TwiML, opens a Twilio Media Streams websocket for call audio, converts inbound Twilio audio frames into bounded turn audio for `VoicePipeline.handle_turn`, and returns synthesized assistant speech to the caller using the simplest reliable Twilio-compatible playback path. Use Twilio caller metadata as the preferred caller phone source for prospect capture. Keep automated tests deterministic with mocked Twilio webhook and media events; do not require live Twilio calls, public tunnels, or provider credentials in CI. M11 will not build a browser voice UI, admin UI, CRM workflow, production call routing, durable call recording storage, or full telephony hardening.

Status: Accepted

Group: real-call-voice-transport

Assumptions: A real Twilio inbound call is feasible for the demo environment with a Twilio account, phone number, and either a public tunnel or deployed callback URL. Twilio caller metadata can provide a useful phone number for the prospect write gate, while spoken phone capture remains available when caller ID is unavailable or masked. The M10 turn-based pipeline remains the source of voice-agent behavior, so M11 should focus on transport adaptation rather than changing grounding, property resolution, model response constraints, or prospect write rules. If Twilio streaming audio cannot be reliably returned as live generated audio within the milestone, the implementation may use Twilio-compatible response playback or call-control redirects as long as the caller experiences a real back-and-forth voice call.

Constraints: M11 must reuse `leasing_voice_assistant.voice_pipeline.VoicePipeline` and the M09 conversation session path rather than creating a separate telephony-specific agent. It must preserve the M08 safe write gate, including low-confidence and confirmation behavior. It must not commit Twilio credentials, public tunnel tokens, provider secrets, real caller audio, real phone numbers, or call recordings. It must keep tests offline by mocking Twilio signatures, webhook payloads, websocket messages, provider responses, and audio payloads. It must bound audio buffering and handle malformed, stale, duplicate, or out-of-order transport events without unsafe writes. It must document required Twilio, public URL, model, STT, and TTS settings clearly.

Positions: Position 1: Build a Twilio inbound-call adapter for M11 using FastAPI webhooks, TwiML, Twilio Media Streams, caller metadata, and the M10 voice pipeline. This best satisfies the brief's ideal real-call evidence and the user's clarified preference for an actual Twilio call.

Position 2: Build a browser voice loop first and leave Twilio as optional. This would reduce credential and public-networking risk, but it no longer matches the clarified direction and provides weaker demo evidence.

Position 3: Implement both Twilio and browser transports in M11. This would give a fallback demo path, but it expands the milestone and risks under-testing the real-call path that matters most.

Position 4: Replace the turn-based M10 pipeline with a fully streaming telephony agent. This could improve latency and interruption behavior, but it would bypass a tested contract and introduce concurrency, partial transcript, cancellation, and audio-framing complexity too early.

Position 5: Use Twilio Gather or recorded-call chunks without media streaming. This is simpler than websocket audio streaming, but it is less conversational and weaker against the milestone's streaming-transport expectation.

Argument: Position 1 is recommended because the project now has the right backend boundary for a telephony adapter. The voice pipeline already owns STT, grounded session handling, constrained model phrasing, TTS, confidence propagation, fallback behavior, and safe debug metadata. Twilio integration should therefore be a thin transport layer: answer the call, collect audio, identify the caller, pass an audio turn into the pipeline, play the assistant's response, and keep the same serializable session state for the next turn.

The main trade-off is operational risk. Twilio requires an account, a provisioned number, public HTTPS/websocket callbacks, trial-account constraints, and careful audio format handling. That risk is acceptable for M11 because the assignment values a real call and the user explicitly chose Twilio. The mitigation is to keep local tests mocked and deterministic, document the manual setup, and keep the implementation small enough that failures are diagnosable as transport, provider, or tunnel issues rather than agent-behavior ambiguity.

The second trade-off is interruption handling. True barge-in across telephony media streams requires detecting caller speech while assistant audio is playing, canceling stale synthesis/playback, and preventing late responses from being played after a newer caller turn. M11 should implement practical stale-turn protection and interruption behavior where Twilio event flow supports it, but it should explicitly document any limitation that remains. The core acceptance criterion is a working real voice conversation, not perfect telephony-grade barge-in.

Twilio caller metadata should improve prospect capture by providing the phone number without requiring the caller to speak it. The assistant should still confirm or ask for missing identity details according to the existing write gate, especially when transcript confidence is low or the caller ID is unavailable. No telephony route may write prospects directly.

Implications: M11 implementation should add a Twilio transport module with typed call/session state, inbound webhook handling, TwiML response generation, media-stream event parsing, audio decoding or adaptation, response playback handling, stale-turn protection, and safe call debug metadata. The FastAPI app should expose the minimal routes needed for Twilio voice and media callbacks. Configuration should add Twilio account SID, auth token or signature validation secret, inbound number, public base URL, media stream URL, and any playback/static-audio settings needed by the chosen response path. Tests should cover inbound webhook TwiML shape, caller metadata propagation, media start/media/stop event parsing, one mocked voice turn through the pipeline, low-confidence prospect capture with Twilio caller ID, malformed event handling, provider degradation mapped to caller-safe behavior, and stale response suppression. README and architecture docs should document Twilio setup, local tunnel/deploy requirements, exact environment variables, and the manual verification script for placing a real call.

Related principles: Keep telephony transport separate from conversation logic. Treat Twilio as an adapter around the existing voice pipeline, not as a second agent implementation. Prefer real-call evidence for the demo while keeping automated tests offline and deterministic. Use caller metadata to improve safety, but never bypass confirmation and write-gate rules. Bound audio and session state. Degrade gracefully when STT, model, TTS, Twilio, or public tunnel setup fails. Keep secrets, real caller data, and recordings out of the repository.
