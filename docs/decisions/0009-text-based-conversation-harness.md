# 0009 Text Based Conversation Harness

Issue: M09 needs a developer-facing way to run a complete leasing conversation in text before voice/audio complexity is added. M07 can answer one grounded text turn and M08 can safely capture prospect interest, but there is no session-level loop that preserves answer state, capture state, transcript history, and debug traces across turns. This issue is being addressed now because M09 is the next incomplete milestone and later voice pipeline, browser/telephony integration, end-to-end tests, evaluation, and observability need a stable conversation-session contract.

Decision: Add a deterministic text-based conversation harness for M09 with a reusable session service plus a thin local CLI entrypoint. The session service will accept caller text turns, optional caller metadata, and prior session state; call the existing grounded answer orchestrator and prospect-capture service; return the assistant reply, updated session state, transcript entries, capture/write outcome, and optional safe debug traces. The CLI will be a developer/demo wrapper over that same service, not a separate behavior path. M09 will not add voice/audio transport, browser UI, Twilio integration, real model calls, Strands integration, persistent multi-session storage, or a polished user interface.

Status: Accepted

Group: conversation-harness

Assumptions: A CLI-first harness is sufficient for M09 because the milestone is meant to make complete conversation behavior debuggable before audio. A reusable session service is still needed so later voice and HTTP/browser layers do not duplicate conversation state handling. Session state can remain in memory and serializable for tests; durable storage can wait until a later milestone if a voice transport requires it. The assistant reply can remain deterministic and template-based for now, composed from M07 answers and M08 capture prompts/outcomes. Debug traces are for developers and tests, not caller-facing transcript text.

Constraints: M09 must reuse the existing M07 answer orchestrator and M08 prospect-capture service rather than bypassing their grounding and write-gate behavior. It must preserve property-resolution state, prospect-capture state, and turn history across text turns. It must keep prospect writes gated exactly as M08 defines and must not introduce any new unsafe write path. It must expose tool/write traces safely enough for local debugging while avoiding committed real personal data or secrets. It must run locally without credentials, network calls, live model inference, audio providers, browser permissions, telephony, or external agent frameworks.

Positions: Position 1: Add a reusable in-memory conversation session service plus a thin CLI wrapper. This keeps the milestone small, testable, and useful for both local manual runs and later voice integration.

Position 2: Build only an HTTP API harness on FastAPI. This aligns with the existing FastAPI scaffold and may help a future browser client, but it adds request/session lifecycle questions before the text behavior is proven and is less ergonomic for quick local transcript debugging.

Position 3: Build only a CLI loop that directly wires M07 and M08. This is fastest for manual testing, but without a reusable service it risks diverging from the later voice/session path.

Position 4: Introduce Strands Agents SDK or a model-backed chat loop now. This could make text conversations more natural and closer to a final agent, but it adds provider credentials, prompt validation, and nondeterminism before there is a complete deterministic baseline.

Position 5: Persist every conversation session to SQLite immediately. This would improve restartability and auditability, but it expands schema and privacy concerns beyond the M09 goal. Later observability or voice milestones can add sanitized persistence if needed.

Argument: Position 1 is recommended because M09 is primarily about session continuity and debuggability. A session service can combine the existing turn-level components without changing their contracts: M07 handles grounded answers and property context, while M08 handles identity, intent, confirmation, and safe prospect writes. The CLI then becomes a simple way to exercise that same service locally and save or copy transcript output for milestone evidence.

The main trade-off is that a CLI-first harness is not the final user experience. That is acceptable because the assignment allows implementation to proceed in layers, and the next milestones are explicitly responsible for voice, browser/telephony transport, and demo recording. M09 should optimize for deterministic behavior, traceability, and tests rather than natural voice polish.

The service should treat prospect-capture prompts as higher priority than generic answer text when the caller is in the middle of registering interest. For example, if M08 needs a name, phone number, property clarification, or confirmation, the assistant reply should ask that focused follow-up while preserving any useful grounded answer from M07 in the turn result for debugging. If M08 writes an interest, the reply should acknowledge the registered interest using the resolved property or unit label without exposing unnecessary internal details.

Implications: M09 implementation should add a conversation-session module with typed request/result DTOs, serializable session state, transcript entry records, turn numbering, debug-trace controls, and tests. It should add a CLI entrypoint that starts a local session, accepts user turns, prints assistant replies, optionally prints debug traces, and exits cleanly. Tests should cover a multi-turn property question with context carryover, a complete prospect-capture flow, a confirmation-required flow, an ambiguous property flow, and transcript/debug trace shape. The README and status docs should record the command for running a local text conversation and a short sample transcript or command-output summary.

Prefer deterministic state and inspectable traces before adding model or voice nondeterminism. Keep debug traces separate from caller-facing replies. Avoid durable storage and UI polish until they serve the voice demo path.
