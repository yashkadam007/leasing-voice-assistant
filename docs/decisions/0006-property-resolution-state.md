# 0006 Property Resolution State

Issue: M06 needs a deterministic way to track which property or unit the caller means across conversation turns. M04 can search properties and return structured candidates, and M05 can retrieve source-attributed knowledge snippets, but neither component owns conversational context. This issue is being addressed now because M06 is the next incomplete milestone, and M07 grounded answer orchestration plus M08 safe prospect capture depend on knowing when property context is resolved, ambiguous, or unsafe for writes.

Decision: Add a framework-neutral property-resolution service with explicit, serializable resolution state. The resolver will consume the latest user text, prior resolution state, and database query-tool results, then return a new state containing property and optional unit candidates, confidence level, resolution status, evidence, and a clarification prompt reason when needed. M06 will use deterministic heuristics over the existing database tool contract: exact property mentions can resolve property context, high-confidence single candidates can become probable but not necessarily write-ready, unit hints can narrow to a unit only when the property is resolved or uniquely inferable, and ambiguous or missing evidence must produce a clarification-required state. M06 will not add model calls, prompt routing, grounded answer generation, prospect writes, voice integration, or a full conversation harness.

Status: Accepted

Group: conversation-state

Assumptions: The MVP dataset remains small, with Lakeview Flats and Cedar Park Townhomes as representative properties. Caller turns may mention a property by name, city, address fragment, or unit-level hints such as bedrooms, view, rent, parking, availability, or "the lake-facing one." Pronoun or contextual references such as "that one" may reuse prior resolved context only when the prior state is not ambiguous. Deterministic state transitions are sufficient for M06 and make later orchestration easier to test.

Constraints: M06 must keep property resolution separate from answer composition and prospect writing. It must not treat a merely plausible match as safe for writes. It must preserve enough evidence for tests and later orchestration to explain why a state is resolved or ambiguous. It must run without credentials, network calls, external services, embeddings, or model inference. It must work through existing repository and database-tool boundaries rather than exposing raw SQL or duplicating persistence logic.

Positions: Position 1: Add a deterministic resolution service with explicit state, confidence, evidence, and clarification-required outcomes. This keeps M06 small, testable, and independent from any later agent framework while producing the state M07 and M08 need.

Position 2: Let the future LLM orchestrator resolve property context from prompts and tool results. This may feel natural once M07 exists, but it would make M06 untestable in isolation and would weaken the safety gate before writes.

Position 3: Store active resolution state in SQLite. This could help long-running sessions, but it adds persistence scope before there is a session manager or voice transport and makes unit tests heavier.

Position 4: Implement only stateless property search helpers and let each turn resolve from scratch. This avoids state complexity, but it fails pronoun/context references and does not support conversation carryover.

Position 5: Add semantic matching or embeddings for property and unit references now. This may improve paraphrase handling later, but it introduces provider choices and nondeterminism before the MVP has a text orchestrator or evaluation scenarios.

Argument: Position 1 is recommended because property resolution is both a conversational state problem and a safety problem. The project needs a durable way to say "resolved," "ambiguous," "no match," or "needs clarification" before grounded answers and prospect writes are allowed to rely on a property context. A deterministic service can encode conservative rules, preserve evidence from database tools, and remain easy to verify with focused scenarios.

The main trade-off is that heuristic resolution will not understand every natural phrase. That limitation is acceptable for M06 because the assignment dataset is small and ambiguity should bias toward clarification rather than guessing. Later orchestration can call the resolver with richer normalized text or model-extracted hints, but the resolver should still own the final safety classification.

Confidence should be represented as an explicit enum rather than only a float. Suggested levels are `resolved`, `probable`, `ambiguous`, and `unresolved`, paired with a write-readiness flag or status. Exact single property matches may become `resolved`. Single high-confidence matches may become `probable` or `resolved` only when evidence is strong enough and tests cover the behavior. Multiple possible properties, conflicting hints, or unit hints that match more than one unit must become `ambiguous`. Missing evidence must become `unresolved`. Write readiness should require a resolved property or unit plus later M08 identity and intent checks.

Implications: M06 implementation should add a property-resolution module, typed state and result DTOs, and focused tests. Tests should cover exact property references, context reuse through pronouns, unit-hint narrowing such as "lake-facing one," ambiguous property references, ambiguous unit references, no-match references, and state reset or downgrade when new evidence conflicts with previous context. The architecture documentation should describe resolver state and clarification policy. Requirements traceability should record that M06 provides concrete evidence for FR-04, FR-05, EVAL-03, NFR-03, and NFR-06, while not yet satisfying grounded answer orchestration, prospect capture, or voice behavior.

Related principles: Preserve conservative safety boundaries. Ask for clarification instead of guessing. Make state serializable and inspectable. Keep the resolver independent from model providers and voice transport. Use existing database-tool evidence rather than raw persistence access. Treat property resolution as necessary but not sufficient for prospect writes.
