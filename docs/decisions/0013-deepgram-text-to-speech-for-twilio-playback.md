# 0013 Deepgram Text To Speech For Twilio Playback

Issue: Manual Twilio call testing showed that the assistant can receive caller audio, transcribe speech through Deepgram streaming STT, resolve and answer property questions, and produce safe assistant text, but callers still hear silence because ElevenLabs rejects the selected Twilio-compatible `ulaw_8000` output with `402 payment_required`. The current Twilio Media Streams path can only send outbound audio when the synthesized bytes are raw 8 kHz mu-law. A TTS provider that can return Twilio-compatible audio without a paid ElevenLabs feature is needed for the live-call demo path.

Decision: Use Deepgram Aura text-to-speech as the preferred real TTS provider for the Twilio call path. Configure Deepgram TTS to return `mulaw` audio with `container=none` and `sample_rate=8000`, so the response can be sent directly as Twilio Media Streams outbound media without MP3/WAV transcoding or file headers. Keep the existing fake TTS provider for deterministic tests. Keep the existing ElevenLabs adapter as a fallback provider option, but do not make it the default documented path for Twilio playback while `ulaw_8000` is blocked by account plan limits.

Status: Accepted

Group: twilio-tts-provider

Assumptions: The existing Deepgram API key can be authorized for both streaming STT and Aura TTS, or a separate Deepgram key can be supplied through the same secret setting. Deepgram TTS REST is sufficient for the MVP because the current `VoicePipeline` is turn-based and already waits for full synthesized speech before sending one Twilio media message. Deepgram's documented `mulaw` plus `container=none` plus `sample_rate=8000` output is compatible with Twilio's requirement for raw `audio/x-mulaw` at 8000 Hz. A single provider for STT and TTS reduces setup friction for assignment demo testing.

Constraints: The implementation must not send MP3, WAV, Opus, AAC, or header-bearing audio bytes to Twilio Media Streams as outbound media. It must not commit Deepgram credentials, real caller data, provider payloads, or recordings. It must preserve the provider abstraction in `interfaces.TextToSpeechProvider` and the `SynthesizedSpeech` return contract. It must preserve fake-provider tests and must not require live provider calls in automated validation. It must keep the M08 safe prospect write gate and M09 conversation session unchanged. It should keep the existing ElevenLabs adapter available unless a later ADR removes it.

Positions: Position 1: Use Deepgram Aura TTS with `encoding=mulaw`, `container=none`, and `sample_rate=8000`. This directly matches the Twilio media format, avoids transcoding, and reuses the existing Deepgram account setup already needed for streaming STT.

Position 2: Keep a cheaper ElevenLabs MP3 output and transcode to raw mu-law locally. This avoids changing providers, but it adds an FFmpeg or audio-conversion dependency, increases latency and deployment complexity, and still depends on ElevenLabs for every call turn.

Position 3: Keep ElevenLabs and upgrade to a plan that supports `ulaw_8000`. This is the smallest code change, but it creates a paid-plan dependency for the demo and already failed during manual testing.

Position 4: Use Amazon Polly with `OutputFormat=mulaw` and `SampleRate=8000`. This is technically compatible and has a documented free tier, but it adds a second cloud account, AWS credentials, request signing, and operational setup during an already working Deepgram STT integration.

Position 5: Use Google Cloud Text-to-Speech with `MULAW`. This has a free character allowance for some voices, but documented MULAW output includes a WAV header, which conflicts with Twilio's warning not to include header bytes in media payloads unless we strip headers correctly.

Argument: Position 1 is recommended because the call path's failure is specifically an audio-format and provider-plan mismatch, not a conversation or STT failure. Twilio requires raw 8 kHz mu-law outbound media. Deepgram documents `mulaw` output with `container=none` and `sample_rate=8000`, including a note that `container=none` avoids telephony clicks or static caused by header bytes. That matches Twilio Media Streams more closely than MP3, WAV, or header-bearing mu-law responses. Deepgram is already part of the project for streaming STT endpointing, so using it for TTS reduces credential and account surface.

The trade-off is that Deepgram Aura voice quality and voice-selection semantics may differ from ElevenLabs. For the MVP, a reliable audible answer over a real phone call is more important than maximizing voice naturalness. The assignment expects a natural, non-robotic call, but a clean Deepgram TTS voice is acceptable if latency remains within the two-to-three-second target. Later hardening can revisit streaming TTS or another provider if evaluation shows voice quality is a weak point.

Implementation should add `deepgram` as a valid `LVA_TEXT_TO_SPEECH_PROVIDER` value, add settings for the Deepgram TTS REST URL and voice/model name if the existing `text_to_speech_model` is not sufficient, and implement a `DeepgramTextToSpeechProvider` in `provider_adapters.py`. The adapter should call Deepgram's TTS REST endpoint with the selected voice/model and output query parameters for raw 8 kHz mu-law audio. It should return `SynthesizedSpeech(audio=..., content_type="audio/mulaw;rate=8000" or "audio/x-mulaw;rate=8000", metadata=...)`. The Twilio outbound compatibility guard should accept Deepgram's content type only when it represents raw mu-law at 8000 Hz.

Tests should verify adapter construction fails clearly without a Deepgram API key, the generated request URL includes `encoding=mulaw`, `container=none`, and `sample_rate=8000`, provider HTTP errors include status/body details, and Twilio outbound media accepts the Deepgram mu-law content type while continuing to reject MP3/WAV content. Existing voice-pipeline and Twilio transport tests should continue using deterministic fake providers and should not call the real Deepgram API.

Implications: The live-call README and `.env.example` should document Deepgram as the recommended TTS provider for Twilio playback. `LVA_TEXT_TO_SPEECH_OUTPUT_FORMAT=ulaw_8000` should no longer be presented as the only Twilio path if it is ElevenLabs-specific and plan-gated. The status document should record that manual debugging found ElevenLabs `402 payment_required` for `ulaw_8000` and Groq model `403`, but the conversation/session logic generated correct safe replies. The implementation should remain scoped to TTS playback; it should not change property resolution, answer orchestration, prospect capture, or Twilio turn detection.

Related decisions: ADR 0002 established provider interfaces and deterministic fakes. ADR 0010 established the transport-neutral voice pipeline with TTS behind a provider boundary. ADR 0011 established Twilio Media Streams as the real-call adapter. ADR 0012 established Deepgram streaming STT endpointing for Twilio turn detection. This ADR extends the real-call path by choosing Deepgram for Twilio-compatible TTS playback.

Related requirements: FR-01 requires a real back-and-forth voice-to-voice conversation. FR-11 requires a real inbound call or voice-to-voice alternative. NFR-01 requires natural spoken output, NFR-02 allows two-to-three-second latency, NFR-03 requires maintainable structure, NFR-06 requires automated verification, NFR-07 forbids committed secrets, and NFR-08 requires provider integrations to be testable without real calls.

Related principles: Keep Twilio outbound audio format strict. Prefer direct provider output over local transcoding for the MVP. Avoid paid-plan surprises in the demo path. Keep provider-specific behavior behind adapters. Keep automated tests deterministic and credential-free. Do not bypass the grounded answer/session/write-gate contracts when changing TTS.
