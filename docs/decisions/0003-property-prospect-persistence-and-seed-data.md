# 0003 Property Prospect Persistence And Seed Data

Issue: M03 needs to implement local persistence for properties, units, prospects, and prospect interests before database query tools, property resolution, grounded orchestration, and safe prospect capture can be built. The storage layer must support factual property answers, prospect upsert by phone number, and idempotent interest logging while remaining easy to run from a clean checkout. This issue is being addressed now because M02 established configuration and repository interfaces, and M04 depends on real storage and representative seed data.

Decision: Use SQLite as the local relational database, managed through deterministic SQL migration files and committed JSON seed data. M03 will add schema migrations for `properties`, `units`, `prospects`, and `prospect_interests`, a small migration runner using Python's standard `sqlite3` module, JSON seed files for one or two synthetic properties and units, and repository implementations for property/unit reads plus prospect upsert and interest insertion. The generated SQLite database file will be local runtime state and not committed. Prospect identity will be matched by normalized phone number, and prospect interest creation will be idempotent for the same prospect, source, and target unit or property. Agent orchestration, model tools, knowledge-base retrieval, voice, and confirmation-gate behavior remain out of scope for M03.

Status: Accepted

Group: persistence

Assumptions: SQLite is sufficient for the MVP's one or two properties, local demo, and deterministic tests. The assignment's provided sample listings are not present in the repository, so M03 may create synthetic seed data that covers the required leasing facts without using real personal data. The M02 repository protocols can be implemented by concrete SQLite repositories without changing provider boundaries. Python's standard `sqlite3` module is enough for this milestone; adding an ORM is not required to build the small schema safely. Phone-number normalization can be basic in M03 and refined later if voice or telephony metadata creates additional edge cases.

Constraints: M03 must not implement model-driven agent orchestration, database tools exposed to an LLM, knowledge-base retrieval, property-resolution state, safe write confirmation, or voice transport. The schema must support the fields from `brief.md`: property name/address/city, unit label, bedrooms, bathrooms, square footage, monthly rent, availability date, view, parking, pet policy, amenities, and unit status. Seed data must be synthetic and must not include real prospects or real phone numbers for real people. Setup must be idempotent so tests and clean checkouts can initialize the same database repeatedly. The generated database file must be ignored or placed in a local runtime path. Tests must not require external services or credentials.

Positions: Position 1: SQLite with SQL migrations, JSON seed data, and concrete repository classes. This keeps the MVP local and reproducible, uses a real relational store, supports constraints and transactions, and avoids adding a database server.

Position 2: PostgreSQL with Alembic or another migration tool. This is closer to a production web service and stronger for concurrent writes, but it adds local service setup, credentials, and clean-checkout friction that the assignment does not need for one or two properties.

Position 3: In-memory Python data structures or committed JSON-only storage. This is fastest to implement, but it does not exercise real database reads/writes, constraints, or prospect upsert behavior, and it weakens later tool and safety tests.

Position 4: SQLModel or SQLAlchemy ORM over SQLite. This gives typed models and future portability, but it adds dependency and mapping complexity before the data model has proven it needs ORM features.

Position 5: DuckDB or another embedded analytical database. This is strong for analytics-style reads, but the project needs transactional prospect writes and simple application persistence rather than analytical query features.

Argument: Position 1 is recommended because it gives the project a real database boundary with the least operational overhead. SQLite is included with Python, runs locally without credentials, and is enough for the assignment's deliberately small data model. SQL migration files make the schema reviewable and deterministic, while JSON seed files keep sample property content easy to inspect and modify. Repository classes preserve the M02 interface boundary and let later milestones add model-safe tools without exposing raw SQL to the agent.

The main trade-off is that SQLite is not a final multi-tenant production database and has limited write concurrency compared with Postgres. That is acceptable for the MVP because the expected demo has one local assistant session and a small dataset. If later requirements demand multi-user deployment or stronger concurrency, a future ADR can supersede this choice and migrate the repository layer to Postgres.

Interest idempotency should be enforced in storage rather than relying only on agent behavior. A unique constraint over the prospect, source, and resolved target prevents repeated confirmations or retries from creating duplicate interest rows. Because the brief allows interest in either a unit or a property, the schema should support a nullable `unit_id` and nullable `property_id` with a check requiring at least one target. For M03 seed and tests, unit-level interest should be the default when a specific unit is known; property-level interest remains available for future resolution flows.

Implications: M03 implementation should add migration and seed-data directories, a SQLite connection/setup module, concrete property and prospect repositories, and tests for schema setup, idempotent seed loading, property/unit reads, prospect upsert by phone number, and duplicate-safe interest insertion. Documentation should describe where generated local database files live and how to reset them. Security impact is low because only synthetic seed prospects, if any, are allowed. Testing impact is positive because later milestones can run against an actual relational schema without external services. Operational impact is limited to local file management and ensuring the generated database file is not committed.

Related decisions: ADR 0001 established Python/FastAPI, `uv`, `src/` layout, `pytest`, `ruff`, and `mypy`. ADR 0002 established configuration, repository protocols, and deterministic fakes. Later ADRs for database query tools, property resolution, grounded orchestration, and safe prospect capture will depend on the persistence layer created by this decision.
