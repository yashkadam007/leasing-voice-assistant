# 0007 Grounded Answer Orchestration

Issue: M07 needs the first text-turn agent behavior that can answer caller questions from the existing database tools, knowledge-base retriever, and property-resolution state. M04 exposes structured database facts, M05 exposes source-attributed knowledge snippets, and M06 resolves property and unit context, but there is not yet a component that chooses those tools, combines their evidence, and composes a caller-facing answer. This issue is being addressed now because M07 is the next incomplete milestone and later prospect capture, text harness, voice integration, and evaluation need an inspectable grounded-answer contract.

Decision: Add a framework-neutral deterministic answer orchestrator for M07. The orchestrator will accept a user text turn plus prior property-resolution state, call the property resolver, route answerable questions to database tools and/or the knowledge retriever, compose a concise grounded text response, and return an inspectable turn result containing updated resolution state, selected route, evidence, answer text, and fallback reason when evidence is missing. M07 will not introduce prospect writes, model-provider calls, Strands integration, voice/audio handling, persistent session storage, or a full conversation UI. It will create the text-turn contract that a later LLM or voice session can reuse.

Status: Accepted

Group: agent-orchestration

Assumptions: The MVP data set remains small enough for deterministic routing to cover the milestone scenarios. Caller questions usually contain enough lexical signal to classify common property facts such as rent, bedrooms, view, parking, pet policy, and availability, or broader KB topics such as application process, deposits, lease terms, amenities, and general policies. The assistant can be useful in M07 with template-based composition as long as it is grounded, conservative, and inspectable. A real model provider can be introduced behind the existing `ModelProvider` protocol in a later milestone after deterministic text scenarios exist.

Constraints: M07 must preserve the safety boundary between answers and writes. It must not invent property facts, lease terms, availability, or policy details when DB/KB evidence is absent. It must use the existing database-tool, KB retriever, and property-resolution interfaces rather than raw SQL or direct Markdown parsing. Tool choice and evidence must be visible to tests. Unknown or ambiguous questions must produce a clarification or graceful fallback. The implementation must run locally without credentials, network calls, embeddings, external agent frameworks, or live model inference.

Positions: Position 1: Add a deterministic text-turn orchestrator with explicit routing, evidence, and template-based grounded answer composition. This keeps M07 small, testable, and independent from provider credentials while producing the contract later model and voice work need.

Position 2: Integrate Strands Agents SDK now and expose DB and KB functions as agent tools. This aligns with the assignment's preferred stack and may be useful later, but it adds framework and model-provider decisions before there is a deterministic baseline, and it risks making tests brittle or credential-dependent.

Position 3: Use the existing `ModelProvider` fake/real boundary for answer composition now while deterministic code only gathers evidence. This moves closer to a production agent loop, but M07 would need prompt contracts, model output validation, and hallucination controls before the evidence path is proven.

Position 4: Build only a thin manual harness that calls DB and KB tools directly from tests. This would demonstrate isolated facts but would not create the central orchestration surface needed by prospect capture and voice.

Position 5: Merge database and KB content into one retrieval index and answer from retrieved snippets only. This simplifies routing, but it weakens structured handling of unit facts, availability, rent, and ambiguity, and it conflicts with the existing separation between database-owned facts and document-owned policy content.

Argument: Position 1 is recommended because M07 is primarily about grounded behavior and inspectability, not provider selection. A deterministic orchestrator can encode conservative routing rules, reuse the M06 resolver, preserve evidence from M04 and M05, and make the milestone scenarios easy to verify. It gives later model-backed orchestration a clear baseline: a turn request, a route decision, evidence records, updated resolution state, and an answer or fallback.

The main trade-off is conversational flexibility. A deterministic classifier will not understand every phrasing, and template-based answers will be less natural than an LLM-composed response. That limitation is acceptable for M07 because the assignment's safety bar is higher than the natural-language polish at this stage. Later voice and model milestones can replace or augment answer composition while keeping the same evidence-first contract and tests.

Routing should prefer database tools for structured property/unit facts such as rent, bedrooms, bathrooms, square footage, unit status, availability, parking, view, and pet-policy values stored on units. It should prefer KB retrieval for general leasing policies, application process, deposits, lease terms, and richer descriptions. If a question could require both, the orchestrator should gather both but keep database facts authoritative for unit-level structured fields. If evidence conflicts or is insufficient, the response should say that the assistant cannot confirm the fact and, when relevant, ask a focused clarification question.

Implications: M07 implementation should add an answer-orchestration module with typed request/result DTOs, route classifications, fallback reasons, and tests. Tests should cover rent from DB, property/unit facts from DB, FAQ or application-process answer from KB, unknown answer fallback, ambiguous property clarification, property context carryover, and conflicting fact handling where database facts take precedence for structured unit fields. The orchestrator should expose evidence internally for assertions and future logs but does not need to show citations to the caller unless that keeps the response natural. Architecture and requirements traceability should record that M07 provides concrete evidence for FR-02, FR-03, FR-04, FR-05, FR-10, EVAL-02, EVAL-03, NFR-03, and NFR-06, while still not satisfying prospect capture or voice behavior.
