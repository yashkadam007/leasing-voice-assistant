# 0010 Voice Pipeline

Issue: M10 needs to add the first model-backed audio path on top of the completed text conversation session. M09 can run a complete leasing conversation in text, including grounded answers, property resolution, transcript state, and safe prospect writes, but its caller-facing replies are deterministic templates rather than natural model-backed speech. The assignment explicitly asks for a natural voice agent, permits free-tier speech and model providers, and expects a tool-using agent that decides which tools to use while staying grounded. This issue is being addressed now because M10 is the next incomplete milestone and M11 needs a transport-neutral, provider-backed voice pipeline before exposing the assistant through Twilio or a browser voice loop.

Decision: Add a transport-neutral, turn-based voice pipeline for M10 that connects Deepgram-capable speech-to-text, model-backed grounded reply generation, the existing conversation session service, and ElevenLabs-capable text-to-speech behind provider interfaces. The pipeline will accept an audio input object plus optional caller metadata, transcribe speech into text with confidence metadata, call the M09 session service for state, tool evidence, property resolution, and prospect-write decisions, optionally ask a model provider to rewrite the safe session reply into a more natural grounded voice response, synthesize the final assistant reply into an audio output object, and return transcript text, confidence, assistant text, synthesized speech, updated session state, timing metadata, and safe debug details. M10 will use deterministic fake model/STT/TTS providers for automated tests and may include real provider adapters configured by environment variables, but tests and default local runs must not require real credentials. M10 will not add Twilio, browser microphone UI, real call handling, websocket streaming, public tunneling, or demo recording.

Status: Accepted

Group: voice-pipeline

Assumptions: A turn-based audio pipeline is sufficient for M10 because the milestone is about proving speech input, model-backed natural response generation, and spoken output around the existing agent state, not implementing a final transport. Browser or telephony streaming can adapt to this turn contract in M11, and a later ADR can add streaming or barge-in if the demo path requires it. Deepgram is the preferred first STT adapter and ElevenLabs is the preferred first TTS adapter because they match the user's intended provider choices, but their adapters must remain replaceable through interfaces. The model provider can be selected through existing configuration conventions, and fake model behavior must remain available for CI. Transcript confidence is available from the STT provider or can be supplied by deterministic fakes in tests. The session service remains the source of truth for property resolution, prospect capture, transcript history, tool evidence, and write gating.

Constraints: M10 must reuse `leasing_voice_assistant.conversation_session` as the only path into answer orchestration and prospect capture. The model provider may improve wording, conversational flow, and voice naturalness, but it must not invent property facts, bypass the DB/KB evidence, override clarification requirements, or perform writes directly. It must preserve the M08 rule that low-confidence or garbled transcripts cannot cause unsafe prospect writes. It must keep model, STT, and TTS behind provider protocols and keep CI tests independent of real credentials, network calls, microphones, speakers, Twilio, browser permissions, or live provider SDK behavior. Real provider adapters must fail clearly when credentials are missing and must not be selected by default in tests. It must keep audio payload handling bounded and avoid committing real caller audio, real personal data, provider responses, or secrets. It must expose enough metadata for later latency logging without implementing the full M14 observability layer.

Positions: Position 1: Add a turn-based voice pipeline module using model/STT/TTS provider protocols, Deepgram and ElevenLabs-ready adapters, deterministic fakes, and the M09 session service. This keeps M10 transport-neutral and testable while making the replies more natural and establishing the provider-backed voice contract needed by M11.

Position 2: Add STT and TTS only, keeping deterministic template replies. This is simpler and safer, but it underuses the brief's model-provider allowance and is less likely to satisfy the natural conversational voice expectation.

Position 3: Implement streaming STT, streaming model responses, and streaming TTS now. This would reduce final-demo latency and better match telephony media streams, but it adds concurrency, partial transcript handling, cancellation, and audio framing before a simple provider-backed turn has been proven.

Position 4: Build a browser voice loop directly using Web Speech APIs and browser speech synthesis. This could produce a fast demo path with limited backend audio code, but it would bind voice behavior to one transport, skip Deepgram/ElevenLabs integration, and delay a reusable backend pipeline for tests and possible Twilio integration.

Position 5: Build Twilio media-stream integration now. This gives the strongest real-call evidence, but it combines provider credentials, public tunnel setup, websocket audio formats, STT, model replies, TTS, and call control in one milestone and risks obscuring pipeline bugs behind telephony setup issues.

Argument: Position 1 is recommended because M10 should establish the smallest provider-backed voice boundary around the already-tested conversation state. The completed M09 session service already owns conversation state, transcript recording, answer orchestration, tool evidence, and capture/write decisions. The model should operate after those decisions as a constrained response composer: it receives the safe session reply, evidence summaries, missing-information prompts, and write-gate outcome, then returns a natural voice response that remains faithful to that data. The voice pipeline should therefore compose audio input to transcript, transcript to session turn, safe session result to model-backed voice text, and voice text to synthesized audio.

The main trade-off is nondeterminism. Model-backed wording is better aligned with the brief's naturalness requirement, but it creates hallucination and test brittleness risk. The mitigation is to keep deterministic tool routing, property resolution, capture state, write gating, and fake-model tests as the source of behavioral truth. Automated tests should assert evidence use, write-gate outcomes, and no unsafe writes rather than exact prose from a live model.

The second trade-off is latency. Turn-based audio with STT, model generation, and TTS is less conversational than streaming with interruption support, but the brief accepts two to three seconds of latency if the conversation remains natural, and barge-in is explicitly out of the MVP unless time permits. Starting turn-based also gives M11 a clear integration target: a browser or telephony transport can collect one utterance, call the pipeline, and play one spoken reply.

Transcript confidence should flow into the session layer as a low-confidence marker or structured metadata that the prospect-capture path can interpret conservatively. If the caller says interest details but STT confidence is below the agreed threshold, the assistant should synthesize a confirmation prompt instead of writing. Grounded answer behavior can still proceed when confidence is acceptable enough for a question but not for a write; the exact threshold can be tuned in implementation tests without changing the ADR.

Implications: M10 implementation should add a voice-pipeline module with typed request/result DTOs, a small audio-input abstraction, confidence handling, model-response metadata, timing fields, and tests using deterministic fake model/STT/TTS providers. It should add or extend provider configuration for model, Deepgram STT, and ElevenLabs TTS credentials without requiring those credentials in tests. Tests should cover a normal spoken question producing a transcript, grounded model-backed assistant text, and synthesized speech; a spoken prospect-interest flow with caller metadata; a low-confidence transcript that requires confirmation and does not write; model output constrained by provided evidence; STT failure returning a caller-safe fallback; model failure falling back to the safe session reply; and TTS failure returning assistant text plus an explicit degraded result. README and architecture docs should document the local fake-voice path and identify Deepgram, ElevenLabs, and the selected model provider as optional real adapters for M10/M11 manual verification.

Related principles: Keep audio transport separate from conversation logic. Preserve the existing grounding and write-gate boundaries. Use the model to improve naturalness and turn phrasing, not as an unchecked source of property truth. Prefer deterministic fake-voice and fake-model tests before live provider calls or telephony setup. Treat transcript confidence as safety-critical for writes. Degrade gracefully when STT, model, or TTS fails. Keep caller-facing audio separate from developer debug metadata.
