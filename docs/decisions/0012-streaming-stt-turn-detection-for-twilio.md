# 0012 Streaming STT Turn Detection For Twilio

Issue: M11 exposed the assistant through a real Twilio inbound-call path, but manual call testing showed the turn boundary is wrong. The current Twilio adapter buffers Media Streams audio and invokes `VoicePipeline.handle_turn` only when Twilio sends a `stop` event. In Twilio Media Streams, `stop` means the stream or call is ending, not that the caller has finished an utterance. The call can therefore connect and play the greeting, but the assistant does not naturally respond after the caller speaks during an ongoing call. M11.1 needs a provider-owned speech endpointing signal so the Twilio path can process one caller turn before call termination.

Decision: Add streaming speech-to-text turn detection to the Twilio call path using Deepgram live transcription endpointing. Twilio remains the call-control and audio-transport adapter. It forwards inbound mu-law media frames to a streaming STT session, receives interim and finalized transcript events, accumulates final transcript segments for the current utterance, and triggers one assistant response when the streaming STT provider marks the utterance complete, such as Deepgram `speech_final=true`. The assistant response must reuse the existing conversation session, grounded answer orchestration, safe prospect-capture write gate, constrained model rewrite, and TTS synthesis path. M11.1 may refactor `VoicePipeline` to expose a transcript-to-response helper, but it must not create a second telephony-specific agent or bypass the M08 write gate. Automated tests stay offline with deterministic fake streaming STT events.

Status: Accepted

Group: twilio-streaming-turn-detection

Assumptions: Deepgram live transcription can accept Twilio-compatible 8 kHz mu-law audio over websocket and can emit usable final transcript and endpointing metadata for short caller utterances. The provider endpointing signal is a better caller end-of-turn boundary than any Twilio Media Streams lifecycle event. Existing M10 and M11 provider abstractions can be extended without requiring real provider credentials in tests. A serial, one-response-at-a-time call loop is sufficient for M11.1; full barge-in, cancellation of in-flight TTS, and production telephony concurrency can remain later hardening. Two to three seconds of latency remains acceptable if the call responds after speech without requiring call termination.

Constraints: M11.1 must preserve the M09 conversation session as the source of transcript history, property context, answer orchestration, and prospect-capture state. It must preserve M08 confidence and confirmation behavior for all writes, including low-confidence transcript handling. It must keep Twilio credentials, Deepgram credentials, real phone numbers, real transcripts, audio recordings, and provider payloads out of the repository. It must not replace the existing batch `SpeechToTextProvider` contract for non-Twilio voice turns unless the replacement remains backwards compatible. It must bound audio forwarding, transcript buffers, and per-call session state. It must handle malformed Twilio frames, malformed provider messages, provider disconnects, empty final transcripts, and duplicate endpoint events without unsafe writes or repeated assistant responses.

Positions: Position 1: Use Deepgram live websocket transcription as the Twilio turn detector, add a small streaming transcript provider boundary, and trigger the existing response path on finalized endpoint events. This directly addresses the M11 manual-test failure while keeping Twilio and conversation responsibilities separate.

Position 2: Continue buffering Twilio media and trigger turns on fixed silence or fixed time windows inside the Twilio adapter. This avoids a second websocket provider, but it makes the transport guess speech boundaries, risks clipping callers or waiting too long, and duplicates a capability the STT provider is designed to supply.

Position 3: Use Twilio `Gather` or call-control redirects instead of Media Streams for turn collection. This is simpler operationally, but it weakens the real streaming call architecture already implemented in M11 and would produce a less natural conversation loop.

Position 4: Replace the M10/M11 turn pipeline with a fully streaming realtime agent that streams STT, model tokens, and TTS audio with interruption support. This could reduce latency and support barge-in, but it is a much larger redesign than the bug requires and would bypass stable grounded-answer and write-gate contracts.

Position 5: Build the browser voice fallback now. This could avoid Twilio media-stream complexity, but ADR 0011 selected Twilio for the real-call demo path and the current blocker is specific and fixable in that path.

Argument: Position 1 is recommended because the responsibilities line up cleanly. Twilio should own phone-call setup, caller metadata, inbound and outbound audio frames, and stream lifecycle. Deepgram should own speech recognition and utterance endpointing. The existing session and voice pipeline should own what happens after a caller utterance becomes text: property resolution, grounded tool use, prospect write safety, natural response phrasing, and synthesis. This avoids treating Twilio lifecycle events as speech semantics and avoids rebuilding agent behavior inside the transport layer.

The main trade-off is websocket complexity. M11 already has one websocket for Twilio Media Streams; M11.1 adds another live websocket to Deepgram and must coordinate two event streams. The mitigation is to keep the provider event contract narrow: accept audio bytes, emit typed transcript events, identify final segments, identify utterance completion, surface confidence when available, and expose safe failure states. Tests should mock these events deterministically rather than depending on network timing.

The second trade-off is pipeline shape. `VoicePipeline.handle_turn` currently owns batch STT, session handling, model rewrite, and TTS. Streaming STT means Twilio should not call the batch STT step again after it already has a finalized transcript. M11.1 should therefore extract or add a helper that accepts a `Transcript` plus caller/session metadata and returns the same `VoiceTurnResult` shape, with STT timing and metadata populated from the streaming path. This keeps the response behavior shared while avoiding duplicate transcription.

Endpointing must be conservative. Interim transcripts are useful for diagnostics, but they must not trigger answers or writes. Final transcript segments should be accumulated until the provider emits an utterance-complete signal. Empty or whitespace-only final transcripts should be ignored. Duplicate endpoint events for the same accumulated utterance should not produce duplicate responses. Low-confidence final transcripts should flow into the existing write gate so prospect writes require confirmation instead of being silently trusted.

Implications: M11.1 implementation should add a streaming STT protocol or adapter module with typed events for interim transcript, final transcript, utterance complete, provider error, and session close. It should add a Deepgram live adapter configured for Twilio mu-law 8 kHz audio and endpointing, using explicit settings for API key, websocket URL, model, language, encoding, sample rate, and endpointing options where needed. `TwilioCallManager` or a related Twilio streaming session object should forward inbound `media` payloads to the streaming STT session instead of waiting for `stop`, accumulate finalized transcript text and confidence, call the shared transcript-to-response path when endpointing fires, and return Twilio-compatible outbound `media` and `mark` messages. The M11 `stop` path should close provider resources and may remain as an emergency diagnostic fallback, but it must not be the primary turn trigger.

Tests should cover Deepgram-style interim, final, and speech-final messages; multiple final segments in one utterance; ignored interim-only turns; ignored empty finals; one Twilio media stream response before Twilio `stop`; duplicate endpoint suppression; provider disconnect or malformed provider payload fallback; caller phone propagation into prospect capture; low-confidence transcript confirmation behavior; non-Twilio TTS audio not being streamed as media; and bounded buffer behavior. README, architecture, status, requirements traceability, and the ADR index should explain that Twilio transports audio while Deepgram endpointing defines caller turn completion.

Related principles: Keep speech endpointing out of Twilio lifecycle events. Keep telephony transport separate from conversation logic. Reuse the existing grounded session, model rewrite, TTS, and safe write gate after transcript finalization. Treat interim transcripts as non-authoritative. Prefer deterministic fake streaming tests over live provider calls. Bound buffers and per-call state. Degrade safely on STT, model, TTS, Twilio, Deepgram, or public tunnel failures. Keep secrets, real caller data, and recordings out of the repository.
