# 0008 Safe Prospect Capture

Issue: M08 needs to turn a caller's interest into a database write without registering the wrong person, property, or unit. M03 already provides prospect upsert and idempotent interest logging, M06 provides write-readiness for property and unit resolution, and M07 provides an inspectable text-turn orchestration contract, but no component yet owns caller identity capture, interest-intent detection, confirmation, or the final write gate. This issue is being addressed now because M08 is the next incomplete milestone and later text harness, voice integration, end-to-end tests, evaluation, and observability depend on a safe prospect-capture contract.

Decision: Add a deterministic safe-capture service for M08. The service will accept the latest user text, optional caller metadata such as telephony/browser phone number, prior property-resolution state, prior capture state, and a prospect repository. It will extract or update caller identity fields, detect clear interest intent, decide whether a write is blocked, needs confirmation, or is allowed, and perform the prospect upsert plus idempotent interest logging only after all write-gate checks pass. M08 will not add voice/audio transport, persistent session storage, real model calls, Strands integration, a full conversation harness, or a CRM-style prospect workflow. It will create the capture/write contract that the later text and voice session layers can call.

Status: Accepted

Group: prospect-capture

Assumptions: The MVP can use deterministic text heuristics for names, phone numbers, and interest phrases because the immediate goal is a testable safety gate, not fully natural extraction. Phone metadata from a future telephony or browser session is more reliable than a spoken phone number when present. Matching prospects by normalized phone number remains the storage rule from ADR 0003. Caller name and phone are the minimum required identity fields for M08; email remains optional and should be captured only if naturally supplied. A property-level interest is acceptable when a property is resolved but no unit is resolved, while unit-level interest is preferred when the resolver has a specific unit.

Constraints: M08 must not write unless property or unit resolution is write-ready, caller name is present and plausible, caller phone is present and normalizable, and interest intent is clear or explicitly confirmed. It must treat ambiguous property/unit state, missing identity, unclear intent, and garbled or low-confidence transcript indicators as blocked or confirmation-required states. It must use the existing `ProspectRepository` interface rather than raw SQL. It must keep write decisions inspectable for tests and future logs. It must run locally without credentials, network calls, live model inference, audio providers, or external agent frameworks.

Positions: Position 1: Add a deterministic capture state machine and write gate with explicit outcomes, reasons, confirmation prompts, and repository calls only at the allowed terminal state. This keeps M08 small, safety-focused, and easy to verify before introducing voice transcription uncertainty.

Position 2: Extend the M07 answer orchestrator to also extract identity and write prospects. This reduces the number of modules, but it mixes read-answer composition with side-effecting writes and weakens the safety boundary M07 deliberately preserved.

Position 3: Wait for the later model-backed or voice-backed agent and ask the model to decide when to call a write tool. This may become useful later, but it makes the first write path nondeterministic and hard to test before the voice stack exists.

Position 4: Always require a final explicit confirmation phrase before every write, even when property, identity, and intent are all clear. This is safest, but it may make the conversation unnecessarily stiff. The better MVP rule is to require confirmation when any critical field came from uncertain text or implicit intent, while still allowing direct writes for fully clear text scenarios in tests.

Position 5: Store every capture attempt as an audit record, including blocked attempts. This could help observability, but it expands schema and logging concerns beyond M08. Later observability work can log sanitized write-gate decisions without changing the prospect schema now.

Argument: Position 1 is recommended because M08 is primarily a safety and state-management milestone. A deterministic state machine can make the write gate explicit: resolved target, clear identity, clear intent, confirmation status, and transcript confidence. It can reuse ADR 0003 storage idempotency, ADR 0006 write-readiness, and ADR 0007 turn orchestration without coupling prospect writes to answer composition.

The main trade-off is extraction flexibility. Simple heuristics will miss some natural phrasings and may ask for clarification more often than a model-backed extractor would. That is acceptable for M08 because false negatives are safer than false positives, and later model or voice work can improve extraction behind the same capture contract. The first implementation should bias toward "ask a short follow-up" rather than "write from a guess."

Confirmation should be represented as explicit state rather than just a phrase check. If a caller says "yes" after the assistant has summarized the target and identity, the service can treat that as confirmation for the pending write. If the caller changes the property, unit, name, or phone after the summary, the pending confirmation must be invalidated and recalculated. A repeated confirmation should be safe because ADR 0003 makes interest logging idempotent for the same prospect, source, and target.

Implications: M08 implementation should add a prospect-capture module with typed request/result DTOs, capture state, write-gate outcome enum, blocked/needs-confirmation/allowed reasons, confirmation prompt text, and tests. Tests should cover missing name, missing phone, ambiguous property, unclear intent, garbled transcript markers, explicit confirmation, duplicate phone upsert, duplicate interest idempotency, unit-level target when resolved, and property-level target when only the property is resolved. The module should return enough detail for future conversation harnesses to produce natural follow-up prompts and enough structured detail for future logs, while avoiding accidental exposure of full phone numbers in logs unless explicitly required.

Related principles: Prefer blocked or confirmation-required over unsafe writes. Treat property resolution as necessary but not sufficient for prospect writes. Prefer caller metadata over spoken extraction when available. Keep side effects isolated behind an explicit write gate. Make write decisions inspectable and idempotent. Avoid storing real personal data in committed fixtures or docs.
